# Working Style

This project is also a learning project for recruitment conversations.

## How To Build

- Work in tiny, reviewable steps.
- Prefer one concept per change.
- Explain the reason for each technical choice in practical terms.
- When introducing or explaining a file, start with who or what uses the file.
- For important files, explain four things: who uses it, why it exists, whether it belongs in Git, and what the alternative is.
- Include best-practice context and mention reasonable alternatives.
- Keep explanations direct rather than writing long summaries.
- Leave deeper theory for follow-up questions unless it is needed for the current step.
- Avoid adding large feature areas before the current step is understood.
- When an important concept, terminology point, or interview explanation appears, include a short copy-paste quote that can be moved to a sticky note.

## Current Baseline

- Use `.NET 10 LTS`.
- Use C# 14 through the installed .NET 10 SDK.
- Keep nullable reference types enabled.
- Treat C# warnings as errors.
- Add implementation only after the project shape and tradeoff are clear.

## Interview Angle

Each step should leave behind something explainable:

- What problem did this solve?
- Why is this a common or safe approach?
- What alternative could have been chosen?
- What did we intentionally defer?

Use this format for sticky-note explanations:

> "Short practical explanation in one or two sentences."
